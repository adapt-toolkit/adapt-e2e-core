# Packaging & build targets

The crate builds a `staticlib` + `cdylib` + `rlib`. The consumer links
`libadapt_e2e_core.a` exactly where it links `libsodium`. Per target:

| Target | Rust lane | Status here | Notes |
|---|---|---|---|
| native x86_64/aarch64 (linux/macos/win) | stable `cargo build --release` | ✅ built + tested | staticlib + committed C header |
| **rv32 bare-metal** | `riscv32imac-unknown-none-elf`, nightly `-Zbuild-std` | ✅ **builds** (`scripts/rv32_baremetal_build.sh`) | no OS, no getrandom — the injected-entropy design win |
| ARM iOS | `aarch64-apple-ios` (+ sim, `x86_64-apple-ios`) | ⛔ consumer-side | needs an Apple toolchain (not on this host); `xtask` packages an xcframework of per-arch staticlibs |
| ARM Android | `aarch64-linux-android` (+ armv7/x86_64) via cargo-ndk | ⛔ consumer-side | needs the consumer's Android NDK; static archive per ABI |
| ARM servers | `aarch64-unknown-linux-gnu` | ✅ stable (same as native) | |
| WASM | `wasm32-unknown-emscripten` | ⛔ **deferred** (owner) | ABI-pinned to the consumer's `emcc`; `emsdk` not on this host. The crate is no_std-clean, so it builds consumer-side. `wasm32-unknown-unknown` is a no_std proxy. |
| RISC-V linux | `riscv64gc-unknown-linux-gnu` | stable-ish | std present |

## Consumer integration

- **Header:** `include/adapt_e2e_core.h` is committed (generated by cbindgen; regen
  with `cargo build --features generate-header`). It declares `enum e2e_rc` and the
  10 `e2e_*` functions with a `#pragma once` + C99 + `extern "C"` C++ guard.
- **no_std / bare-metal:** the crate is `#![no_std]` under `--no-default-features`.
  A no_std *library* defers the global allocator, `#[panic_handler]`, and
  `panic = "abort"` to the consumer's firmware/link — the rlib/staticlib build
  verifies the crate's code is no_std-correct.
- **xtask packaging** (xcframework / cargo-ndk) is straightforward Rust cross-builds
  once the Apple/Android toolchains are present; it is not runnable on this
  linux-only host, so it is left as a documented consumer/CI step.

## Pinned forks (re-audit: see PATCH.md)

Two vendored, security-backport-only forks: `vendor/vodozemac` (the `*_with_rng`
seam + no_std/getrandom-severance delta) and `vendor/base64` (unconditional
`core::error::Error` for no_std). Both verified behaviour-preserving.
